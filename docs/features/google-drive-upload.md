# Google Drive Upload — Architecture Documentation

> Generated by feature-documentation-architect · 2026-05-16

---

## 1. Executive Mental Model

Google Drive import is a **server-proxied ingestion path** that lets a photographer select files already stored in Google Drive and bring them into the Wave Atlas media pipeline without downloading and re-uploading locally.

The feature has two primary moving parts:

- **Client-side orchestration** (`useGooglePicker`): acquires a short-lived OAuth2 token, opens the Google Picker UI, and fires one tRPC mutation per selected file
- **Server-side ingestion** (`registerDriveImport → CloudinaryService → MediaRepository`): fetches the file from Google's CDN using the user's own access token, uploads it to Cloudinary, and persists a draft `MediaItem`

The architecture deliberately keeps the OAuth token **short-lived and client-scoped**: the server never stores it. It only uses it once — to fetch the file bytes from Drive — then discards it.

```
┌─────────────────────────────────────────────────────────────────┐
│  BROWSER                                                        │
│                                                                 │
│   AddSourceCard ──► useGooglePicker                             │
│                          │                                      │
│                    1. OAuth2 token (GIS)                        │
│                    2. Google Picker UI                          │
│                    3. tRPC mutation (per file)                  │
│                          │                                      │
└──────────────────────────┼──────────────────────────────────────┘
                           │  registerDriveImport {
                           │    remoteFileId,
                           │    mimeType,
                           │    accessToken
                           │  }

│  SERVER (tRPC + Express)                                        │
│                                                                 │
│   mediaRouter ──► MediaService.registerDriveImport              │
│                          │                                      │

└─────────────────────────────────────────────────────────────────┘
```

The design philosophy: **the client is the gatekeeper of identity, the server is the gatekeeper of storage**. The client proves Drive access via OAuth; the server translates that into a durable, Wave-Atlas-owned Cloudinary asset.

## 2. Problem & Design Motivation

### The core problem

Photographers have archives in Google Drive — thousands of images and videos. Requiring them to download and re-upload locally is friction that kills conversion. The feature eliminates the download step.

### Why not have the browser upload directly?

```
Simpler rejected path:
  Browser ──► Drive API (download file bytes) ──► Cloudinary (upload bytes)
```

This would work for small images but has three critical problems:
1. **CORS**: Cloudinary's upload endpoint requires a signed server-generated signature. The signature includes folder, type, and eager transforms — it can't be generated client-side without exposing the API secret.
2. **Memory**: Video files (up to 50MB) held in browser memory while simultaneously downloading from Drive and uploading to Cloudinary is fragile on mobile.
3. **Token exposure**: The OAuth access token would be visible in network logs going to Cloudinary's servers.

### The chosen architecture: server-proxied ingestion

```
Browser sends: { remoteFileId, mimeType, accessToken }
Server does:   fetch(driveUrl, { Authorization: Bearer token })
               → cloudinary.uploader.upload(driveUrl, { headers })
```

Cloudinary's `upload_from_url` accepts custom HTTP headers, which means the server can tell Cloudinary to fetch the file directly from Google's CDN using the user's token. **The bytes never touch the server** — it's a Cloudinary→Google transfer, brokered by a server-side instruction.

| Concern | Solution |
|---|---|
| API secret exposure | Server-only; never sent to browser |
| Eager transforms | Applied at upload time server-side, same as client path |
| Token lifetime | OAuth token is single-use; server never stores it |
| File size | Cloudinary handles the fetch; no server memory pressure |

## 3. Core Conceptual Model

### Domain concepts

| Concept | Owner | Lifecycle | Invariant |
|---|---|---|---|
| **OAuth2 access token** | Browser (GIS) | Acquired on demand, used once, discarded | Never persisted; expires in ~1h |
| **remoteFileId** | Google Drive | Permanent within Drive | Opaque string 10–100 chars alphanumeric |
| **UploadItem** | Zustand | Created on picker selection; destroyed on queue clear | Never holds MediaItem data directly |
| **QueueItem** | Derived (render-time) | Computed from Zustand + TanStack Query | Never stored in either store |
| **MediaItem (DRAFT)** | TanStack Query + DB | Created by registerDriveImport; transitions to PUBLISHED | Always owned by a single photographer |
| **Cloudinary asset** | Cloudinary + DB publicId | Created with eager transforms; deleted if DB write fails | `authenticated` type — never directly accessible |
| **importingItems** | Local state (useGooglePicker) | Skeletons created on picker confirm; removed on mutation settle | Parallel to actual QueueItems — merges at render |

### Concept relationship map

```
                     ┌─────────────────┐
                     │   Google Drive  │
                     │  (OAuth2 scope) │
                     └────────┬────────┘
                              │ access_token + remoteFileId
                     ┌────────▼────────┐
                     │ useGooglePicker  │
                     │ (orchestration)  │
                     └────────┬────────┘
                              │ registerDriveImport mutation
              ┌───────────────▼───────────────┐
              │         mediaRouter            │

              └───────────────┬───────────────┘
                              │
          ┌───────────────────▼───────────────────┐

          └──────────┬─────────────────────────────┘
                     │                     │
```

### Responsibility matrix

| Module | Owns | Must not do |
|---|---|---|
| `useGooglePicker` | OAuth flow, picker UI, skeleton management | Know about Cloudinary or storage |
| `mediaRouter` | Rate limiting, input validation, auth context | Business logic |
| `MediaService` | Import orchestration, ownership, cleanup | Direct Cloudinary SDK calls |
| `CloudinaryService` | Cloudinary API abstraction | Know about media domain or DB |
| `MediaRepository` | DB persistence | Business rules |
| `useUploadQueue` | Merge Zustand + TQ into QueueItem[] | Store data or trigger mutations |

## 4. Runtime Story *(most important)*

### Happy path: user imports 2 photos from Drive

```
sequenceDiagram
  actor User
  participant AddSourceCard
  participant useGooglePicker
  participant GIS as Google Identity Services
  participant Picker as Google Picker UI
  participant tRPC
  participant MediaService
  participant Cloudinary
  participant DB as PostgreSQL
  participant TQ as TanStack Query

  User->>AddSourceCard: clicks "Google Drive"
  AddSourceCard->>useGooglePicker: trigger()
  useGooglePicker->>GIS: initTokenClient({ scope: drive.readonly })
  useGooglePicker->>useGooglePicker: setIsPickerLoading(true)
  GIS-->>useGooglePicker: callback(accessToken)
  useGooglePicker->>Picker: PickerBuilder.build().setVisible(true)
  useGooglePicker->>useGooglePicker: setIsPickerLoading(false)

  User->>Picker: selects photo_a.jpg + photo_b.jpg → PICKED

  useGooglePicker->>useGooglePicker: setImportingItems([skeleton_a, skeleton_b])
  Note over useGooglePicker: Cards appear immediately as "importing"

  loop for each doc (sequential)
    useGooglePicker->>tRPC: registerDriveImport({ spotId, remoteFileId, mimeType, accessToken })
    tRPC->>MediaService: registerDriveImport(userId, input)
    MediaService->>Cloudinary: uploadFromUrl(driveUrl, { Authorization: Bearer })
    Cloudinary->>Google Drive: GET /files/{id}?alt=media (with token)
    Google Drive-->>Cloudinary: file bytes
    Cloudinary-->>MediaService: { publicId, thumbnailUrl, lightboxUrl }
    MediaService->>DB: createMedia({ status: DRAFT, ... })
    DB-->>MediaService: MediaItem
    MediaService-->>tRPC: MediaItem
    tRPC-->>useGooglePicker: MediaItem
    useGooglePicker->>TQ: append(mediaItem)
    useGooglePicker->>useGooglePicker: remove skeleton from importingItems
  end
```

### Failure path: Drive import fails (network / Cloudinary error)

```
useGooglePicker->>tRPC: registerDriveImport(...)
tRPC->>MediaService: registerDriveImport(...)
MediaService->>Cloudinary: uploadFromUrl(...)
Cloudinary--X MediaService: error

MediaService: (no DB write — publicId doesn't exist yet)
MediaService-->>tRPC: throws
tRPC-->>useGooglePicker: catch block

useGooglePicker->>useGooglePicker: notify.error("Failed to import photo_a.jpg")
useGooglePicker->>useGooglePicker: remove skeleton_a from importingItems (finally)
Note: photo_b.jpg import continues
```

### Failure path: DB write fails after successful Cloudinary upload

```
MediaService->>Cloudinary: uploadFromUrl(...) → OK (publicId acquired)
MediaService->>DB: createMedia(...) → throws (DB error)

MediaService->>Cloudinary: deleteAsset(publicId) [fire-and-forget]
Note over MediaService: Cleanup is async — logged on failure, never re-thrown
MediaService-->>tRPC: re-throws DB error
```

### State lifecycle of a single import

```
User selects file
       │
       ▼
importingItems: [{ id: 'drive-import-{docId}', status: 'importing' }]
       │
       ▼  registerDriveImport resolves
TanStack Query cache: append(mediaItem)
importingItems: skeleton removed
       │
       ▼
useUploadQueue merges:
  serverOnlyDrafts includes the new MediaItem
  → QueueItem { status: 'completed', result: MediaItem }
       │
       ▼
UploadGallery renders DRAFT card
```

## 6. File & Directory Reasoning

### Orchestration (Drive-specific)

| File | Why it exists | What breaks if removed |
|---|---|---|
| [useGooglePicker.ts](src/features/Upload/model/useGooglePicker.ts) | Encapsulates the entire OAuth2 + Picker flow. Isolated so the Drive concern doesn't bleed into `useUploadManager` | Drive import entry point disappears |
| [AddSourceCard.tsx](src/features/Upload/ui/cards/AddSourceCard.tsx) | Only UI surface where Drive import is triggered | No UI affordance for Drive |

### Server ingestion (Drive-specific)

| File | Why it exists | What breaks if removed |
|---|---|---|
| [media.ts](src/server/routes/media.ts) — `registerDriveImport` | Rate-limited tRPC procedure; separates auth/validation from business logic | No server endpoint; Drive imports unauthenticated |
| [MediaService.ts](src/server/services/MediaService.ts) — `registerDriveImport` | Orchestrates Drive URL construction, Cloudinary delegation, DB save, orphan cleanup | Caller would need to know Cloudinary + DB details |
| [CloudinaryService.ts](src/server/services/CloudinaryService.ts) — `uploadFromUrl` | Abstracts Cloudinary SDK; applies same eager transforms as client-side path | Drive imports wouldn't get thumbnails/watermarked previews |

### State & queue (shared with local upload)

| File | Why it exists | Role in Drive path |
|---|---|---|
| [useUploadQueue.ts](src/features/Upload/model/useUploadQueue.ts) | Merges Zustand + TanStack Query | Makes Drive-imported items appear after `append()` without page refresh |
| [useDraftMedia.ts](src/features/Upload/model/useDraftMedia.ts) | TanStack Query cache for server-side drafts | `append()` optimistically inserts the Drive-imported MediaItem |
| [uploadStore.ts](src/features/Upload/model/uploadStore.ts) | Zustand pipeline state | Drive imports **bypass** this store — tracked in `importingItems` local state |
| [types.ts](src/features/Upload/model/types.ts) | `UploadStatus` includes `'importing'` — Drive-specific status | Without `importing`, skeleton cards have no status to render |

## 7. State Management & Data Flow

### Two parallel state channels

The Drive import flow creates **temporary local state** (`importingItems`) that is *parallel* to, not inside, the Zustand store. This is intentional.

```
CHANNEL A — importingItems (local React state in useGooglePicker)
  Purpose: show skeleton cards while mutations are in-flight
  Lifecycle: created on picker PICKED, destroyed per-file as mutations settle
  Never reaches TanStack Query or Zustand

CHANNEL B — TanStack Query cache (useDraftMedia)
  Purpose: authoritative list of server-persisted MediaItems
  Lifecycle: populated by append(mediaItem) on mutation success
  Feeds into useUploadQueue → UploadGallery
```

### Why `importingItems` instead of Zustand?

Drive imports don't go through the upload pipeline (`UploadPipeline.ts`). There's no `signing → uploading → saving` progression. The import is a single atomic server operation. Adding a skeleton to Zustand would require the store to know about Drive — a coupling leak.

```
Local uploads:
  Zustand UploadItem: pending → signing → uploading → saving → completed

Drive imports:
  importingItems skeleton: { status: 'importing' }  ──► removed on settle
  TQ cache: append(MediaItem { status: DRAFT })
```

### Source of truth map

| Data | Source of truth | Where else it appears |
|---|---|---|
| OAuth access token | Browser memory (never persisted) | Sent in mutation payload; used once server-side |
| Import in-flight status | `importingItems` local state | Rendered as skeleton card |
| MediaItem content (price, URLs) | PostgreSQL via TanStack Query | TQ cache (client), `result` on QueueItem |
| Cloudinary asset | Cloudinary CDN | `publicId`, `thumbnailUrl`, `lightboxUrl` in DB |

## 8. Error Handling Philosophy

### Error origin map

```
Google Identity Services
  └─► response.error → notify.error("Google Drive access was denied")
      isPickerLoading reset to false

Cloudinary uploadFromUrl
  └─► throws → MediaService re-throws
      No DB write occurred → no cleanup needed

DB createMedia (after successful Cloudinary upload)
  └─► throws → MediaService catches
      cloudinary.deleteAsset(publicId) [fire-and-forget, async]
      logs cleanup failure — never blocks the re-throw
      re-throws DB error

tRPC re-throw
  └─► useGooglePicker catch block
      notify.error(`Failed to import "${doc.name}"`)
      skeleton removed via finally block
      Loop continues to next file
```

### Principles

- **Fail per-file, continue the loop** — one failure never aborts remaining imports
- **Orphan cleanup is async, non-blocking** — the original error is what the caller sees
- **User-facing errors are named** — include the filename so the user knows which file failed
- **Access token never logged** — only in Authorization header; scrubbed by log aggregators

## 9. Extension & Maintenance Guide

### Safe extension points

| Task | Where | Notes |
|---|---|---|
| Add allowed MIME type | `PICKER_MIME_TYPES` in `useGooglePicker.ts` + `registerDriveImportSchema` | Both must change together |
| Add another cloud source (Dropbox) | New `useDropboxPicker.ts` + new tRPC procedure | `CloudinaryService.uploadFromUrl` already accepts arbitrary URL + headers |
| Increase rate limit | `driveImportLimiter` in `media.ts` | Currently 20/min |
| Surface import progress | `uploadFromUrl` would need streaming | Cloudinary doesn't expose fetch progress — currently atomic |

### Dangerous coupling areas

- **`importingItems` vs Zustand `uploadQueue`** — Drive skeletons are local state, not Zustand. If someone adds Drive items to `uploadQueue`, they'll appear twice in the gallery. The separation is load-bearing.
- **`append()` optimistic update** — if a `draftMedia` refetch occurs before the append settles, the appended item is overwritten. Don't rely on it for anything beyond rendering.
- **Cloudinary `type: 'authenticated'`** — all assets use authenticated delivery. If this changes, watermarking logic breaks.

### Hidden assumptions

- `accessToken` has `drive.readonly` scope — enforced by `initTokenClient`, not the server
- Cloudinary's `upload()` from URL respects the `Authorization` header — true today, undocumented
- `remoteFileId` regex `[a-zA-Z0-9_-]{10,100}` — based on observed Drive ID format

### Anti-patterns

- Don't add Drive items to Zustand's `uploadQueue` — they don't follow the pipeline lifecycle
- Don't store `accessToken` in Zustand, TQ, or localStorage
- Don't retry `registerDriveImport` automatically — the access token may have expired

## 10. Ultra-Condensed Cheat Sheet

### What this feature does
Server-proxied Google Drive import: browser gets OAuth token → server tells Cloudinary to fetch from Drive → saves DRAFT MediaItem.

### Architecture bullets
- `useGooglePicker` → GIS OAuth2 → Google Picker → `media.registerDriveImport` (tRPC)
- Server: `mediaRouter` (rate limit) → `MediaService` → `CloudinaryService.uploadFromUrl` → `MediaRepository.createMedia`
- Skeleton state: `importingItems` (local) ≠ Zustand `uploadQueue` (pipeline)
- MediaItem lands in TanStack Query via `useDraftMedia.append()`

### Dependency map
```
AddSourceCard
  └─ useGooglePicker
       ├─ google.accounts.oauth2 (GIS)
       ├─ google.picker (Picker)
       ├─ trpc.media.registerDriveImport
       └─ useDraftMediaMutate.append

media.registerDriveImport
  └─ MediaService.registerDriveImport
       ├─ CloudinaryService.uploadFromUrl ──► Cloudinary (fetches from Drive CDN)
       └─ MediaRepository.createMedia     ──► PostgreSQL
```

### Runtime flow (one file)
```
click Drive → OAuth token → Picker opens → user selects
→ skeleton card shown
→ registerDriveImport { remoteFileId, mimeType, accessToken }
→ server: tell Cloudinary to fetch driveUrl with Bearer
→ Cloudinary stores asset, returns URLs
→ DB createMedia (DRAFT)
→ append(MediaItem) to TQ cache
→ skeleton removed
→ DRAFT card visible in gallery
```

### Critical invariants
1. `accessToken` never stored — used once server-side
2. DB write fails after Cloudinary upload → `deleteAsset()` called (async)
3. Drive imports bypass Zustand `uploadQueue` — use `importingItems` local state
4. All assets use Cloudinary `type: 'authenticated'` — originals never public
5. Import loop is per-file — one failure doesn't abort remaining imports

### Debugging entry points

| Symptom | Where to look |
|---|---|
| Picker doesn't open | `useGooglePicker.trigger()` → check `window.google?.accounts?.oauth2` is loaded |
| OAuth error | GIS callback `response.error` field |
| Import fails silently | `media.ts` rate limiter, `registerDriveImportSchema` validation |
| Card doesn't appear | `useDraftMedia.append()` → `useUploadQueue` merge logic |
| Orphaned Cloudinary asset | `MediaService` catch block + `CloudinaryService.deleteAsset` logs |
| Wrong MIME type rejected | `registerDriveImportSchema` mimeType pattern in `mediaSchemas.ts` |
